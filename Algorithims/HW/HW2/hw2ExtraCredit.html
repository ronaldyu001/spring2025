Insertion Sort Comparisons Merge Sort Comparisons
Analysis: In order to understand how the time efficiency would scale with each input size, the worst and average number of comparisons for each input size were analyzed. It was assumed that the time efficiency would scale proportional to the number of comparisons (more interested on average case). A run time of one microsecond was assumed for each comparison.

Looking at the INSERTION SORT, there is an average of 249,750 comparisons for an input size of 1,000. At 1 microsecond per comparison, the estimated time efficiency is ~.25 seconds. This ended up being incorrect, with an actual time efficiency of .01.

Although the time per comparison was off, the scaling of the number of comparisons between each input size can be looked at. A breakdown of how the comparison scaled between input sizes is shown below: 
     1,000 ->    10,000:   ~100 (Actual:  ~95)
    10,000 ->   100,000:   ~100 (Actual: ~100)
   100,000 ->   250,000:     ~6 (Actual:   ~6)
   250,000 ->   500,000:     ~4 (Actual:   ~4)
   500,000 -> 1,000,000:     ~4 (Actual:   ~4)

Taking the above information into account, the time efficiencies for the insertion sort seems to have scaled proportional to the average number of comparisons. Only the 1,000 to 10,000 seemed to be off. Although the number of comparisons scaled by ~95, the time efficiency scaled by only ~43, which was quite the discrepancy. This may be due to the 1,000 and 10,000 input sizes being able to fit inside the CPU cache, thus allowing for faster memory access. The 100,000 and up data sizes may be too large to fit in the CPU cache, and was accessed through RAM instead. Although if this is the case, it was expected that the time efficiency would scale larger than the comparisons when looking at 10,000 to 100,000 input sizes, which was not seen.

Looking at the MERGE SORT. As expected, the time efficiency also scaled proportional to the number of comparisons. However, it had the same discrepancy (time efficiency scaled significantly smaller than comparisons scaled), this time between the 1,000 to 10,000 AND 10,000 to 100,000 input sizes, possibly for the same reason (cache vs RAM).
